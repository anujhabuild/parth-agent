02

The loop — the beating heart

This is the one idea worth understanding. The agent doesn't answer in a single shot. It circles: think, pick tools, use them, read what happened — then asks itself "am I done?" If not, it goes round again. Watch the glowing request travel the track.

03

How it uses tools (safely)

Tools are the agent's hands. It's careful about which it picks up — and the order it uses them in.

It only carries what it needs. Before every step a tiny router reads the last few messages and looks for clues — words like "search", "click", "remember" — then includes just those tool groups. The file tools come along every time; the rest are invited only when relevant. select_tools()

All at once — when it's safe

Reading files, searching code, looking things up. These don't step on each other, so they run together for speed.

read_file
search_code
glob_files

One-by-one — when it's risky

Editing files, running commands, clicking the screen, calling outside servers. These change things, so they wait their turn.

edit_file
run shell

04

What makes it feel smart

Six pieces quietly shape every answer — all stored on your own machine.

Agents

markdown personas

Swap the assistant's "personality" — a code reviewer, a researcher, a setup helper. Each is just a markdown file whose instructions slot into the brain.

Skills

SKILL.md packs

Reusable playbooks the agent loads by itself when the task matches — like this very design guide. Write once, reuse forever.

Memory

facts about you

Your name, preferences and decisions are remembered between chats, so you never repeat yourself.

MCP servers

outside connections

Bridges to the wider world — GitHub, ClickUp, Notion and more — appear as extra tools when connected.

Project context

CLAUDE.md / PARTH.md

If a project has its own rulebook, the agent reads it automatically and plays by those rules.

Sessions

SQLite history

Every conversation is saved locally so you can search, revisit and pick up exactly where you left off.
